# testing-and-feedback.md — 검증 명령과 피드백 루프

> PR 전에 이 문서의 명령을 전부 통과시킨다. 이 문서는 실행 명령의 **단일 출처(SSOT)** 다.
> 하네스 원칙: 에이전트는 "잘 됐다"고 말로 주장하지 않고, 명령을 실행해 증명한다.

## 검증 명령 (스캐폴드 직후 실제 명령으로 갱신 — 첫 번째 작업)

| 목적 | 명령 | 상태 |
|---|---|---|
| 의존성 설치 | `pnpm install` | 예정 |
| 개발 서버 | `pnpm dev` | 예정 |
| 린트 | `pnpm lint` | 예정 |
| 타입 검사 | `pnpm typecheck` | 예정 |
| 단위 테스트 | `pnpm test` | 예정 |
| 프로덕션 빌드 | `pnpm build` | 예정 |
| 데이터 파이프라인 | `pnpm build:data` | 예정 |
| 백테스트 리포트 | `pnpm backtest` | 예정 |

- '상태' 열은 스캐폴드 후 `동작` 으로 바꾼다. 명령이 바뀌면 이 표부터 고친다.
- CI(GitHub Actions): `lint + typecheck + test + build` 를 PR마다 실행 — 설정 후
  이 줄을 실제 워크플로 파일 경로로 교체한다.

## 단위 테스트 최소 범위 (이것만은 반드시)

1. `lib/prediction/models.ts` — 각 모델의 수치 예시 고정 테스트(스냅샷 아님, 값 비교).
2. 도달 예상일 계산 — 경계 사례: 추세 0/양수(→안정), 임계 정확히 걸침, 31일 초과.
3. `lib/data` 정규화 — 결측/이상치/보간 규칙.
4. 가뭄단계 판정 — 70/60/50/40 경계값 5단계 전부.

UI 스냅샷 테스트는 하지 않는다(2주 프로젝트 유지비 > 효용). 대신 아래 수동 QA.

## 수동 QA 시나리오 (릴리스 전 체크리스트)

- [ ] 신규 사용자 풀 플로우: 스플래시 → 온보딩 → 동의 → 주소 검색 → 등록 → 메인
- [ ] 4개 상태(정상/가뭄/심각/장마) 각각: 게이지·도달 카드·그래프·코치·배너 정합
- [ ] 지역 2개 이상 등록 → 전환/삭제(선택 지역 삭제 포함) → 메인 반영
- [ ] 로딩: 메인 스켈레톤 / 검색 스피너 / 버튼 스피너 / rainfall 각각 노출
- [ ] API 강제 실패(키 제거)에서 폴백 문구 노출, 화면 불사
- [ ] 375px(iPhone SE)·큰 글꼴 설정에서 깨짐 없음
- [ ] `prefers-reduced-motion`에서 모든 애니메이션 정지
- [ ] 예측 문구가 전부 '참고' 표현인지 (확정 표현 grep: "됩니다"·"위험합니다" 검색 0건)

## 관측성 (에이전트 셀프 디버깅 경로)

- Vercel 배포 로그·런타임 로그로 API Route 오류를 확인한다.
- `lib/data` 페치 실패는 콘솔에 구조화 로그(`{source, status, fallback}`)로 남긴다 —
  에러를 삼키지 않는다. 침묵 실패는 하네스의 적이다.

## 성능 가드

- 초기 JS 번들 200KB(gzip) 이하 목표. 초과 시 원인 의존성을 tech-stack.md 금지 목록과
  대조하고 제거를 우선 검토한다.
